## ADR D-4 — Estrutura de Repositórios: Monorepo vs 2 Repositórios

```yaml
title: "Unificar admin+web em monorepo vs manter 2 repositórios separados"
status: proposed
date: 2026-07-03
owner: CTO

context: |
  O projeto Meia-Entrada possui duas aplicações Next.js — `up/admin` (painel interno
  de gestão de entidades, contratos e solicitações LGPD) e `up/web` (portal público
  de emissão de carteirinhas, validação e exercício de direitos do titular). Ambas
  compartilham tipagens, schemas Zod, utilitários Supabase, componentes de UI e
  lógica de assinatura CIE v3.3. Atualmente, cada aplicação vive em seu próprio
  diretório dentro do mesmo repositório Git (`up/`), mas com `package.json`, 
  `next.config.mjs`, `middleware.ts` e `tsconfig.json` independentes. Não há
  pacote compartilhado formal — o compartilhamento é feito por referência relativa
  ou duplicação.

  O roadmap de adequação LGPD + CIE v3.3 exige:
  - Pacote de assinatura criptográfica CIE (consumido por admin ao assinar e por
    web ao validar)
  - Tipagens compartilhadas para `lgpd_consents`, `lgpd_subject_requests`,
    `lgpd_contracts`, schema `carteiras` estendido
  - Utilitários de sessão e middleware que já precisam ser idênticos em ambos

  Sem uma estrutura definida, o risco de deriva entre as duas bases é alto,
  especialmente durante sprints S1-S2 com entregas paralelas.

decision: |
  Manter 2 aplicações separadas (admin e web) no mesmo repositório, cada uma
  com seu `package.json` e configuração, mas extrair o código compartilhado
  para um pacote interno (`packages/shared/`) versionado por workspace.
  Inicialmente via npm workspace (sem ferramenta extra); evolui para Turborepo
  se o número de pacotes crescer (ex.: pacote de assinatura CIE, pacote de
  validação offline, pacote de componentes UI). A unificação completa em
  single-page monorepo (ex.: Next.js App Router com rotas `/admin/*` e
  `/*` no mesmo processo) fica postergada para depois da estabilização
  da CIE v3.3 (pós-S5).

consequences:
  - positive:
      - Código compartilhado tem versão única, eliminando deriva de tipos
      - CI/CD unificado com dependências atômicas (build shared → build admin+web)
      - Refatorações que afetam ambos os lados são feitas em um único PR
      - Custo zero de adoção: npm workspaces já são suportados pelo toolchain
      - Baixo atrito para devs (mesmo repo, mesma branch)
      - Postergar monorepo total evita retrabalho durante sprints críticos S1-S2
  - negative:
      - npm workspaces têm limitações de hoisting e resolução de dependências
      - Duas inicializações Next.js em dev (dois processos `next dev`)
      - Deploy separado por app (admin em rota privada, web em rota pública)
      - Se no futuro houver 3+ apps, a falta de Turborepo causará ineficiência
        de cache de build
  - neutral:
      - CI precisa ser adaptado para build sequencial (shared → admin → web)
      - Documentação do workspace deve ser adicionada ao `README.md`

options:
  - id: "D-4-a"
    title: "Manter 2 repositórios totalmente separados"
    description: >
      admin e web vivem em repositórios Git distintos. Código compartilhado é
      duplicado ou publicado como pacote npm privado.
    pros:
      - Isolamento total de deploys e CI
      - Times podem evoluir independentemente (não se aplica hoje)
      - Superfície de ataque menor para admin (repo privado separado)
    cons:
      - Duplicação de tipos e utilitários gera deriva inevitável
      - CI/CD duplicado para mesma stack
      - Mudanças que afetam ambos requerem PRs sincronizados
      - Aumento de carga cognitiva para dev único (Davi)

  - id: "D-4-b"
    title: "Monorepo completo com um único Next.js app"
    description: >
      Unificar admin e web em uma única aplicação Next.js com App Router,
      usando layout/rota para separar `/admin/*` do front público.
    pros:
      - Build único, deploy único, processo dev único
      - Compartilhamento total de tipos, componentes, middleware e configuração
      - Menor complexidade de CI/CD
    cons:
      - Risco de segurança: admin exposto na mesma build que o front público
      - Next.js middleware teria que lidar com roteamento condicional
      - Migração disruptiva durante sprints S1-S2 de compliance
      - Variáveis de ambiente separadas (admin precisa de mais privilégios) seriam
        misturadas
      - Bloqueador de aceitação: tempo de migração incompatível com prazos LGPD

  - id: "D-4-c"
    title: "2 apps no mesmo repo + pacote interno compartilhado (escolhido)"
    description: >
      Estrutura com npm workspaces: `packages/shared` com tipos, schemas Zod,
      utilitários Supabase e lógica comum. admin e web importam via `@meia-entrada/shared`.
      CI faz build do shared primeiro, depois admin e web em paralelo.
    pros:
      - (vide positive consequences acima)
    cons:
      - (vide negative consequences acima)

recommendation: |
  Opção D-4-c — 2 apps no mesmo repo + pacote interno compartilhado via npm
  workspaces. É a de menor esforço de migração (dias), compatível com o sprint
  S2 de governança, e desbloqueia a criação do pacote de assinatura CIE
  (necessário em S3-S4) sem a sobrecarga de um monorepo completo. Reavaliar
  Turborepo ou Nx quando houver 3+ pacotes ou times múltiplos.
```

## ADR D-5 — Prazos de Retenção por Categoria de Dado

```yaml
title: "Prazos de retenção por categoria de dado pessoal"
status: proposed
date: 2026-07-03
owner: DPO / Jurídico

context: |
  A plataforma Meia-Entrada coleta e processa dados pessoais em múltiplas
  categorias com diferentes sensibilidades e finalidades:

  1. **Dados cadastrais** (nome, CPF, RG, data de nascimento, filiação,
     e-mail, telefone, instituição de ensino) — finalidade: emissão da
     carteirinha e vínculo com a entidade emissora.
  2. **Foto biométrica** — dado sensível (art. 11 LGPD), finalidade:
     identificação visual na carteirinha CIE.
  3. **Logs de acesso e operação** (SELECT em PII, criação/alteração de
     carteiras, exercício de direitos, alteração de consentimento) —
     finalidade: auditoria, segurança e resposta a incidentes.
  4. **Consentimentos** — registros de aceite/rejeição com IP, user-agent
     e versão da política.

  A Lei Geral de Proteção de Dados (art. 15) determina eliminação dos dados
  após o fim da finalidade de tratamento, e a ANPD orienta que prazos sejam
  definidos por categoria. Atualmente não há política formal de retenção;
  os dados são mantidos indefinidamente.

  A ausência de prazos configura risco de violação aos arts. 15 e 16 LGPD
  (eliminação desnecessária vs retenção excessiva), além de exposição
  desnecessária em caso de incidente de segurança.

decision: |
  Estabelecer os seguintes prazos de retenção por categoria:

  | Categoria | Prazo de retenção ativa | Prazo de eliminação | Justificativa |
  |---|---|---|---|
  | **Foto biométrica** | Validade da carteirinha + 1 ano | Hard-delete até 90 dias após fim da retenção ativa | Dado sensível; art. 11 exige mínimo necessário; 1 ano extra para contestações de autenticidade |
  | **Dados cadastrais** | Enquanto titular tiver vínculo ativo com alguma entidade emissora + 1 ano | Hard-delete até 90 dias após fim da retenção ativa | Necessário para reemissão e verificação de vínculo; 1 ano extra para obrigações legais |
  | **Logs de acesso a PII** | 5 anos corridos | Hard-delete até 90 dias | Prazo prescricional cível (art. 205 CC) e ANPD (art. 52 LGPD — 5 anos para multa) |
  | **Logs de operação (não PII)** | 2 anos | Hard-delete até 90 dias | Apenas para depuração técnica; sem conteúdo pessoal |
  | **Consentimentos** | 5 anos após revogação ou fim do vínculo | Hard-delete até 90 dias | Necessário para comprovar base legal (ônus do controlador, art. 8 §6º) |
  | **Registros de direitos do titular** | 5 anos após resolução | Hard-delete até 90 dias | Comprovação de cumprimento de prazos e obrigações (art. 18/19) |

  O hard-delete é executado por job agendado (cron mensal) que consulta uma
  view de "expurgo" e elimina registros das tabelas originais, mantendo apenas
  entrada anonimizada no `access_audit_log` (sem PII) para fins de auditoria
  de eliminação.

consequences:
  - positive:
      - Conformidade com arts. 15 e 16 LGPD (eliminação após fim da finalidade)
      - Redução do volume de PII armazenado, mitigando risco de incidentes
      - Prazo de 5 anos para logs alinhado ao prazo prescricional ANPD
      - Hard-delete com janela de 90 dias permite rollback em caso de erro
      - Job de eliminação é auditável e rastreável
  - negative:
      - Requer implementação de job cron (Supabase Edge Function ou pg_cron)
      - Requer view de expurgo e testes de integridade referencial
      - Titulares com vínculo ativo contínuo nunca têm dados eliminados
        (efeito colateral aceitável: finalidade ainda vigente)
      - Prazo de 90 dias para hard-delete pode ser questionado pela ANPD
        como retenção residual excessiva (mitigado por ser janela técnica)
  - neutral:
      - Política de retenção deve ser publicada em `/privacidade` (L-03)
      - Job de eliminação deve ser documentado no RDM (08-rdm-registro-operacoes.md)
      - Revisão anual dos prazos pelo DPO + Jurídico

options:
  - id: "D-5-a"
    title: "Prazo único para todos os dados (ex.: 5 anos)"
    description: >
      Manter todos os dados pessoais por 5 anos independentemente da categoria,
      eliminando todos em lote único.
    pros:
      - Implementação mais simples (uma regra, um job)
      - Menos risco de eliminar algo incorretamente
    cons:
      - Viola princípio da necessidade (art. 27 LGPD) para foto biométrica
      - Mantém dado sensível por tempo excessivo (risco desproporcional)
      - Não atende à orientação ANPD de prazos por categoria

  - id: "D-5-b"
    title: "Retenção por categoria com prazos variáveis (escolhido)"
    description: >
      Prazos específicos para cada categoria conforme tabela na decisão.
    pros:
      - Conformidade granular com art. 15
      - Mínimo necessário para cada finalidade
      - Prazo de logs alinhado à prescrição ANPD
    cons:
      - Complexidade de implementação (múltiplos jobs/queries)
      - Necessidade de view de expurgo com joins entre tabelas
      - Risco de erro em um dos prazos levar a retenção indevida

  - id: "D-5-c"
    title: "Retenção indefinida com hipótese de eliminação manual"
    description: >
      Manter todos os dados até que o titular exerça direito de eliminação
      (art. 18, IV) ou o DPO decida caso a caso.
    pros:
      - Zero implementação técnica inicial
      - Dados disponíveis para qualquer consulta futura
    cons: |
      - Violação direta ao art. 15 (dados mantidos após finalidade)
      - Risco jurídico extremo em auditoria ANPD
      - Depende de ação do titular, transferindo ônus indevido
      - Inviável para foto biométrica (dado sensível sem prazo)

recommendation: |
  Opção D-5-b (prazos por categoria). A implementação é de médio esforço
  (3-5 pontos de história) e pode ser entregue dentro do sprint S2 de
  governança. A recomendação do plano (foto: 1 ano após validade, cadastro:
  ativo + 1 ano, logs: 5 anos) é mantida e expandida para cobrir todas as
  categorias de dados existentes. A janela de hard-delete de 90 dias é um
  compromisso técnico aceitável, documentado na política de retenção e no RDM.
```

## ADR D-6 — Modelo de DPO: Centralizado vs Por Entidade vs Híbrido

```yaml
title: "Modelo de DPO (Encarregado pelo Tratamento de Dados Pessoais)"
status: proposed
date: 2026-07-03
owner: CEO / Fundador

context: |
  O art. 41 da LGPD exige que o controlador indique um encarregado (DPO)
  pelo tratamento de dados pessoais. A ANPD, por meio da Resolução CD/ANPD
  nº 2/2022 (Regulamento de DPO), estabelece que:

  - O DPO pode ser pessoa física ou jurídica (interna ou terceirizada)
  - O DPO pode atuar para mais de um controlador, desde que não haja conflito
    de interesses
  - O DPO deve ser identificado publicamente e estar acessível aos titulares
    e à ANPD

  No modelo de negócio da Meia-Entrada, as entidades emissoras (faculdades,
  escolas, associações estudantis) atuam como co-controladoras (art. 5, IX,
  c/c art. 42-A) — isto é, compartilham as decisões referentes ao tratamento
  dos dados dos seus estudantes.

  Três modelos são possíveis:

  1. **DPO centralizado (Meia-Entrada):** um único DPO indicado pela
     Meia-Entrada como controladora principal, atende todas as entidades e
     titulares.
  2. **DPO por entidade:** cada entidade emissora indica seu próprio DPO,
     responsável pelos dados dos seus estudantes. Meia-Entrada tem seu DPO
     para dados operacionais.
  3. **Modelo híbrido:** um DPO central na Meia-Entrada para questões
     transversais (política, segurança, ANPD) + contato delegado por
     entidade para questões locais (consentimento de menor, eliminação
     de cadastro).

  A escolha impacta diretamente o canal de direitos do titular (F-01), o
  contrato de co-controladora (F-03), o custo operacional e a agilidade
  de resposta.

decision: |
  Adotar o modelo híbrido: um DPO central (pessoa jurídica terceirizada ou
  funcionário dedicado) indicado pela Meia-Entrada como encarregado principal
  para todos os fins do art. 41, publicado em `/privacidade` e no canal
  de direitos do titular. Cada entidade emissora indica um **contato local
  de privacidade** (não um DPO substituto), que atua como ponte entre o
  DPO central e a entidade para:

  - Coleta de consentimento de responsável de menor (F-02)
  - Verificação de identidade do titular em pedidos de direitos (F-01)
  - Encaminhamento de comunicados da ANPD
  - Apoio em incidentes de segurança envolvendo dados daquela entidade

  O contato local não substitui o DPO central — não responde à ANPD nem
  toma decisões sobre bases legais ou prazos de retenção. É um papel
  operacional, não de governança.

  Mecanismo:
  - Campo `contato_privacidade` na tabela `entidades` (e-mail e telefone)
  - O contrato de co-controladora (subprompt 04) obriga a entidade a manter
    o contato atualizado
  - O DPO central tem autonomia para escalar para o representante legal
    da entidade se o contato não responder em 48h

consequences:
  - positive:
      - Um único ponto de contato para a ANPD e titulares (simples e compliant)
      - DPO central garante consistência de respostas e prazos (art. 18/19)
      - Entidades não precisam contratar DPO próprio (barreira de entrada menor)
      - Contato local reduz atrito operacional para questões presenciais
      - Custo previsível (1 DPO central ao invés de N DPOs)
  - negative:
      - DPO central precisa conhecer o contexto de cada entidade (carga cognitiva)
      - Contato local pode ser pessoa sem treinamento em privacidade
        (exige cláusula contratual de capacitação)
      - Entidade pode tentar usar contato local como "DPO paralelo"
        (exige governança clara no contrato)
      - DPO central terceirizado precisa de NDA e cláusula de confidencialidade
        com cada entidade
  - neutral:
      - Nomeação do DPO central deve ser formalizada em ata de reunião
      - Publicação no site (nome ou razão social, e-mail, endereço) conforme
        art. 41 §1º e Res. CD/ANPD nº 2/2022

options:
  - id: "D-6-a"
    title: "DPO centralizado (Meia-Entrada como único DPO)"
    description: >
      Meia-Entrada nomeia um DPO que responde por todos os dados da plataforma.
      Entidades não têm papel de privacidade.
    pros:
      - Simplicidade máxima
      - Um canal, uma política, uma resposta
      - Custos reduzidos
    cons:
      - DPO central não tem capilaridade para verificar consentimento de menor
        presencialmente
      - Entidades podem se sentir excluídas da governança
      - Risco de o DPO não conhecer especificidades locais de cada entidade

  - id: "D-6-b"
    title: "DPO por entidade"
    description: >
      Cada entidade emissora indica e custeia seu próprio DPO, responsável
      pelos dados dos seus estudantes. Meia-Entrada tem DPO apenas para
      dados operacionais.
    pros:
      - Especialização local
      - Entidades têm controle direto sobre privacidade dos seus dados
      - Alinhado com modelo de co-controladoria estrita
    cons: |
      - Custo proibitivo para entidades pequenas (DPO terceirizado custa
        R$ 1.500-5.000/mês)
      - Fragmentação de respostas: titular com dados em múltiplas entidades
        teria que contatar N DPOs
      - Complexidade de coordenação para incidentes transversais
      - Risco de conflito de interpretação entre DPOs sobre mesma base legal
      - Aumento de churn de entidades (barreira de entrada)

  - id: "D-6-c"
    title: "Modelo híbrido (escolhido)"
    description: >
      DPO central Meia-Entrada + contato local por entidade (delegado
      operacional). Conforme detalhado na decisão.
    pros:
      - Melhor relação custo-benefício
      - Um DPO responsável perante ANPD + capilaridade local
      - Entidade não precisa de DPO próprio, mas tem papel ativo
    cons:
      - (vide negative consequences)
      - Requer contrato bem redigido (subprompt 04) para evitar ambiguidade
        de responsabilidades

recommendation: |
  Opção D-6-c (híbrido). É a recomendação explícita do plano de ação
  (seção 4, D-6). Atende ao art. 41 sem criar barreira de entrada para
  entidades pequenas e sem fragmentar a resposta ao titular. O contato
  local é um papel operacional de baixo custo (treinamento de 4h via
  capacitação online), e o DPO central garante consistência jurídica e
  prazos. Recomenda-se que o DPO central seja pessoa jurídica terceirizada
  especializada em LGPD para educação (custo estimado: R$ 2.000-3.500/mês),
  enquanto o custeio pode ser rateado entre as entidades via taxa de
  adesão (decidir em D-7 ou em pricing).
```

## ADR D-7 — Atendimento a Entidades sem CNPJ

```yaml
title: "Atendimento a entidades emissoras sem CNPJ"
status: proposed
date: 2026-07-03
owner: CEO / Fundador + Jurídico

context: |
  A plataforma Meia-Entrada permite que entidades estudantis (faculdades,
  escolas, associações, diretórios acadêmicos, grêmios, coletivos) se
  cadastrem como emissoras de carteirinhas. No modelo LGPD, essas entidades
  são co-controladoras (art. 42-A c/c art. 5, IX), corresponsáveis pelos
  dados pessoais dos estudantes que emitem.

  O contrato de co-controladora (subprompt 04) exige uma pessoa jurídica
  formal para assinar o instrumento e assumir responsabilidade legal.
  No entanto, parte das entidades estudantis (especialmente grêmios,
  diretórios acadêmicos e coletivos) **não possui CNPJ próprio** —
  funcionam como grupos informais ou têm CNPJ da instituição de ensino
  mantenedora.

  As alternativas são:

  1. **Atender sem CNPJ:** permitir cadastro com termo de responsabilidade
     assinado por pessoa física (presidente do grêmio, coordenador de curso).
  2. **Só com CNPJ:** exigir CNPJ próprio ou da mantenedora + procuração
     para assinar o contrato.
  3. **Não atender:** recusar entidades sem personalidade jurídica formal.

  Essa decisão é gate para o onboarding de entidades (F-03) e impacta
  diretamente o modelo de negócio.

decision: |
  Atender apenas entidades com **CNPJ próprio ou da mantenedora**,
  acompanhado de procuração ou documento que comprove vínculo e autoridade
  para assinar o contrato de co-controladora. Entidades sem nenhum vínculo
  com pessoa jurídica (grupos informais sem CNPJ da mantenedora) não são
  atendidas neste momento.

  Exceção limitada: associações estudantis em fase de formalização podem
  obter um cadastro provisório de **90 dias**, mediante termo de
  responsabilidade do representante legal da pessoa física, enquanto
  regularizam o CNPJ. Findo o prazo, se não houver contrato assinado,
  o cadastro é desativado e as carteiras emitidas no período são mantidas
  (já que o titular é o estudante, não a entidade).

  Mecanismo:
  - Campo `cnpj` + `tipo_documento` (próprio / mantenedora / formalizando)
    na tabela `entidades`
  - Upload de documento comprobatório (cartão CNPJ, contrato social,
    procuração, estatuto registrado)
  - Validação automática de CNPJ via API ReceitaWS (ou similar) com fallback
    manual para análise do DPO
  - Flag `contrato_vigente` no gate de emissão (F-03) só true se
    `tipo_documento != 'formalizando'` OU se o prazo de 90 dias não expirou
  - Cadastro provisório tem `validade_experimentacao` e notificação automática
    15 dias antes do vencimento
  - Se a entidade não regularizar, o contato local (D-6) é notificado e o
    cadastro é desativado — carteiras emitidas permanecem válidas

consequences:
  - positive:
      - Contrato de co-controladora com contraparte com personalidade jurídica
        (segurança jurídica)
      - Redução drástica do risco de co-controladoria sem capacidade de
        assumir obrigações legais
      - Alinhado com a recomendação do plano de ação (seção 4, D-7)
      - Janela de 90 dias permite capturar entidades em formalização sem
        perder o relacionamento
      - Carteiras emitidas não são revogadas (proteção ao titular), mesmo
        que a entidade não regularize
  - negative:
      - Exclusão de grêmios, coletivos e movimentos estudantis informais
        (potencial perda de volume de emissão)
      - Necessidade de validação de CNPJ + procuração (custo operacional)
      - Cadastro provisório requer monitoramento ativo (risco de esquecimento
        e manutenção indevida)
      - Entidade mantenedora (faculdade) pode se recusar a assinar contrato
        por grêmio, forçando o grêmio a buscar CNPJ próprio
  - neutral:
      - A política deve ser comunicada claramente no onboarding
      - Revisão semestral do impacto de exclusão (quantas entidades perdidas?)
      - Se o volume de entidades sem CNPJ for significativo (>20% do pipeline),
        reabrir ADR com alternativa de associação formalizadora terceira

options:
  - id: "D-7-a"
    title: "Atender entidades sem CNPJ com termo de responsabilidade"
    description: >
      Entidade sem CNPJ pode se cadastrar mediante termo de responsabilidade
      assinado por pessoa física (presidente, coordenador), que assume
      solidariamente as obrigações de co-controlador.
    pros:
      - Máximo alcance de entidades (grêmios, coletivos, diretórios)
      - Baixa barreira de entrada (crescimento rápido)
      - Alinhado com modelo atual (informal)
    cons: |
      - Risco jurídico: pessoa física pode não ter patrimônio para responder
        por violação LGPD (art. 42)
      - Termo de responsabilidade pode ser questionado pela ANPD como
        insuficiente para co-controladoria
      - Dificuldade de cobrar contratualmente obrigações de DPO, políticas
        e notificação de incidentes
      - Aumento de risco de churn por incapacidade de cumprir obrigações

  - id: "D-7-b"
    title: "Atender apenas com CNPJ (escolhido)"
    description: >
      Exigir CNPJ próprio ou da mantenedora para cadastro como entidade
      emissora, com exceção temporária de 90 dias para formalização.
      Conforme detalhado na decisão.
    pros:
      - Segurança jurídica na co-controladoria
      - Contraparte com personalidade jurídica e capacidade de assumir
        obrigações
      - Alinhado com requisitos de contrato (subprompt 04)
    cons:
      - Exclusão de entidades informais
      - Necessidade de validação documental (custo operacional)
      - Janela de 90 dias demanda monitoramento

  - id: "D-7-c"
    title: "Não atender entidades sem CNPJ (restritivo)"
    description: >
      Bloquear totalmente o cadastro de entidades sem CNPJ próprio.
      Sem exceções ou janela de formalização.
    pros:
      - Zero risco de co-controladoria frágil
      - Máxima simplicidade operacional
    cons: |
      - Perda de volume relevante de emissão (grêmios, diretórios)
      - Exclusão do público-alvo original da Meia-Entrada
      - Risco de migração de entidades para concorrentes
      - Contraditório com a missão de inclusão do estudante

recommendation: |
  Opção D-7-b (só com CNPJ ou da mantenedora + janela de 90 dias para
  formalização). É a recomendação explícita do plano de ação e equilibra
  segurança jurídica com flexibilidade operacional. A janela de 90 dias
  é essencial para não perder entidades que estão em processo de
  formalização — dado que muitas associações estudantis levam 30-60 dias
  para obter CNPJ. A revisão semestral do impacto (quantas entidades
  perdidas vs risco mitigado) permite reabrir a decisão se o custo de
  exclusão for maior que o benefício.
```
