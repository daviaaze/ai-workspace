## Priority 1 ‚Äî Critical

### C1: `gtk_button_set_child` assertion failures

**Problem:**  
`Gtk-CRITICAL **: gtk_button_set_child: assertion 'child == NULL || priv->child == child || gtk_widget_get_parent (child) == NULL' failed`

The same widget is being set as a child of a `Gtk.Button` while still parented to another widget. This happens when Gnim re-renders a button's child content and the old child widget hasn't been unparented.

**Affected locations:**

| File | Component | Pattern |
|------|-----------|---------|
| `src/widget/common/actionButton.tsx` | `ActionButton` | `<Gtk.Button><Gtk.Box>‚Ä¶</Gtk.Box></Gtk.Button>` |
| `src/widget/bar/indicators/bluetoothAudio.tsx` | bluetooth battery indicator | `<Gtk.Button><Gtk.Box><For>‚Ä¶</For></Gtk.Box></Gtk.Button>` ‚Äî `<For>` re-renders can reparent children |
| `src/widget/dock/item.tsx` | DockItem | Multiple `<Gtk.Button><Gtk.Box>‚Ä¶</Gtk.Box></Gtk.Button>` inside popover |
| `src/widget/quicksettings/expander/media.tsx` | PlaybackButtons | `<Gtk.Button><Gtk.Image ‚Ä¶/></Gtk.Button>` ‚Äî button with child icon |

**Fix approach (3 sub-tasks):**

1. **`actionButton.tsx`** ‚Äî Use `Gtk.Button`'s built-in properties instead of a child `Gtk.Box`:
   - Set `iconName` and `label` directly on `Gtk.Button` (GTK4 supports both).
   - If the layout needs both icon+label, use a `Gtk.Box` as child but ensure it's created once, not on every render (Gnim re-render issue).

2. **`bluetoothAudio.tsx`** ‚Äî The `<For>` inside a `Gtk.Button > Gtk.Box` is re-rendering the child box children. Wrap the `Gtk.Button` content in a container that's created once. Or restructure to use `Gtk.Button` with `icon-name` and `tooltip-markup` directly, removing the child `Gtk.Box`.

3. **`dock/item.tsx`** ‚Äî Popover buttons with `Gtk.Box` children can reuse the `ActionButton` component after fixing it, or use label/icon properties directly.

4. **`media.tsx`** ‚Äî Playback buttons: `<Gtk.Button><Gtk.Image ‚Ä¶/></Gtk.Button>` ‚Äî the `Gtk.Image` is being reparented. Create child widgets once, or use `Gtk.Button` with `icon-name` property.

**Verification:** Run `pnpm run build` + observe `journalctl --user _COMM=shade-shell --boot 0 -n 500 --no-pager | grep button_set_child` ‚Äî should be zero.

### C2: `AdwToggleGroup` ‚Äî toggle name not found

**Problem:**  
`Adwaita-CRITICAL **: Trying to set an active toggle with the name '6512ef98f800' in an AdwToggleGroup, but such a toggle does not exist`

The name `'6512ef98f800'` is a hash auto-generated by Gnim or GTK. `Adw.Toggle` components are missing explicit `name` properties, and the `AdwToggleGroup` tries to activate a toggle by a name that doesn't match.

**Affected locations:**

| File | Toggle Group | Toggles |
|------|-------------|---------|
| `src/widget/settings/general.tsx` | Theme toggle (Auto/Light/Dark) | `<Adw.Toggle label={‚Ä¶} iconName={‚Ä¶} />` ‚Äî **no `name` prop** |
| `src/widget/settings/bar.tsx` | Position toggle (Top/Left/Right/Bottom) | `<Adw.Toggle name={‚Ä¶} ‚Ä¶ />` ‚Äî names are numeric strings from `Astal.WindowAnchor` enum values |

**Fix:**

1. **`general.tsx`** ‚Äî Add explicit `name` properties to each `Adw.Toggle`:
   ```tsx
   <Adw.Toggle name={"auto"} label="Auto" iconName="night-light-symbolic" />
   <Adw.Toggle name={"light"} label="Light" iconName="weather-clear-symbolic" />
   <Adw.Toggle name={"dark"} label="Dark" iconName="weather-clear-night-symbolic" />
   ```
   Then update the `onNotifyActiveName` handler to use these names instead of `active` index:
   ```tsx
   onNotifyActiveName={(self) => {
     const map: Record<string, number> = { auto: 0, light: 1, dark: 2 }
     settings.setColorScheme(map[self.activeName] ?? 0)
   }}
   ```

2. **`bar.tsx`** ‚Äî Already has `name` props but they're numeric strings. Verify the `activeName`/`onNotifyActiveName` usage matches. The names `0`, `1`, `2`, `3` should correspond to `TOP`, `LEFT`, `RIGHT`, `BOTTOM` enum values.

**Verification:** Open Settings ‚Üí General ‚Üí click theme toggle. Open Settings ‚Üí Bar ‚Üí click position toggle. No CRITICAL in logs.

## Priority 2 ‚Äî High

### H1: "Managed devices" GtkStack duplicate (every 10 min)

**Problem:**  
`Gtk-WARNING **: While adding page: duplicate child name in GtkStack: Managed devices`

Repeats every ~10 minutes. Coincides with periodic NetworkManager scans. The string "Managed devices" is an internal Astal or Adw widget name for a stacked page that gets added twice during refresh cycles.

**Root cause investigation needed first ‚Äî three possible sources:**

1. **AstalNetwork / NM client refresh** ‚Äî `src/widget/settings/network.tsx` creates `knownNetworks` via `getKnownNetworks(network.client)`. When NM refreshes (every ~10 min by default), the `PreferencesPage` may re-add its internal stack pages.

2. **Avahi/DNS-SD device discovery** ‚Äî Astal's internal device monitoring may manage visible network devices (wifi, ethernet, bluetooth) via a stack with named pages.

3. **Settings window internal Adw stack** ‚Äî The `Adw.PreferencesPage` uses a `Gtk.Stack` internally. If sections are dynamically added/removed, names might collide.

**Fix plan:**

1. **Investigate**: Wrap the settings network section in a guard that checks if widgets already have parents before adding.
2. **Mitigation**: If the Astal internal stack is the source (no control over library code), suppress the warning by setting an error handler:
   ```ts
   // In App.tsx or main.ts
   GLib.log_set_writer_func((logDomain, logLevels, message) => {
     if (message?.includes("duplicate child name in GtkStack")) {
       return GLib.LogWriterOutput.HANDLED
     }
     return GLib.LogWriterOutput.UNHANDLED
   })
   ```

**Verification:** Run for 15+ minutes, grep for "Managed devices" ‚Äî zero occurrences.

### H2: Settings GtkStack duplicates on open

**Problem:**  
`Gtk-WARNING: While adding page: duplicate child name in GtkStack: Help / Notifications / Urgency / Zoom / Settings / Debug`

Appears every time the Settings window is shown (because `hideOnClose` preserves the old widget tree, and Gnim re-renders add new pages alongside old ones).

**Root cause:** The `createSettingsWindow()` function in `src/widget/settings/index.tsx` creates an `<Adw.Window hideOnClose>`. When `window.present()` is called again (e.g., via `shade-shell toggle settings`), the same window is re-shown, but Gnim re-evaluates the widget tree and calls `add` on the internal `Gtk.Stack` with the same page names.

**Fix:**

Option A (preferred): **Destroy and recreate settings window** ‚Äî Remove `hideOnClose`, let the window be destroyed on close. On next open, create fresh.

Option B: **Guard against duplicate adds** ‚Äî Before adding children, check if a child with the same name already exists in the parent. This requires wrapping the `Adw.PreferencesPage` children with a unique key check.

Option C: **Use `set_visible` instead of `present`** ‚Äî If the window is already visible, don't re-render the content.

**Verification:** Open Settings, close, open again ‚Äî no GtkStack warnings.

### H3: `GLibUnix.signal_add_full` deprecation

**Problem:**  
`Gjs-WARNING: GLibUnix.signal_add_full has been renamed. Please update your code to use GLibUnix.signal_add instead.`

**Location:** `src/main.ts:26`

**Fix:** Replace `signal_add_full` with `signal_add`. The new API signature:

```ts
// Old (deprecated):
GLibUnix.signal_add_full(GLib.PRIORITY_DEFAULT, sig, callback)

// New:
GLibUnix.signal_add(GLib.PRIORITY_DEFAULT, sig, callback)
```

The parameters are the same ‚Äî only the function name changed.

**Verification:** Rebuild and run ‚Äî no deprecation warning on startup.

## Priority 3 ‚Äî Medium

### M1: Fingerprint `start failed` errors

**Problem:**  
`Gjs-Console-CRITICAL: [Shade] [ERROR] [fingerprint] start failed: {"stack":‚Ä¶}`

Repeats 3√ó on every lock screen (MAX_RETRIES). The `start()` method catches D-Bus errors and logs them at ERROR level. This is noisy but not breaking ‚Äî errors like `fprintd` not running, device busy, or no finger sensor are expected.

**Fix:**

1. **Reduce log level** ‚Äî Change `logger.error("fingerprint", "start failed:", e)` to `logger.warn("fingerprint", "start failed:", e)` ‚Äî this is expected behavior when fprintd isn't running.

2. **Check fprintd availability first** ‚Äî In `start()` and `init()`, check if fprintd service is running on D-Bus before attempting calls:
   ```ts
   // Before starting, verify the fprintd service exists
   const bus = Gio.DBusProxy.new_for_bus_sync(...)
   // or just check NameHasOwner
   ```

3. **Log less on consecutive failures** ‚Äî After the first retry failure, only log at DEBUG level for subsequent retries.

**Verification:** Lock screen ‚Üí fingerprint retries should be logged at WARN level, not ERROR/CRITICAL.
